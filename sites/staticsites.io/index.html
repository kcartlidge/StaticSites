How Static Sites can help

  • Run all your static websites on a single port with just one server.
  • Have your sites running HTTPS with absolutely zero configuration needed.
  • No need for a traditional web server deploy, meaning tiny resource usage.
  • Production ready with a single file deployment and great performance.
  • Open source codebase for peace of mind: view the code on my GitHub.

Installing Static Sites

Whether you are wanting to run on your local machine or a server/cloud instance makes no difference. Follow the steps below to get started.

Download the relevant version

  • Download the Mac OS version
  • Download the Linux version
  • Download the Windows version

Check the dependencies and install it

There are no dependencies. There is no installation needed.
Just drop the download into a suitable folder and it is ready to run.

Running Static Sites

Create a sites.ini file alongside the application. Run the following, replacing example.com with your site's domain:

cd <download-folder>
echo "example.com = example.com" > sites.ini

Copy your static site into place in a folder alongside the downloaded application, named as per the domain. You can then run the server:

./StaticSites -local example.com -sites sites.ini -port 8000

This will launch all sites listed in the sites.ini file on port 8000 (which is the default, but shown here for completeness). The sites will all start responding to requests to their domain name provided of course that your DNS routes it in.

The local parameter allows you to nominate a site to run as localhost so there is no need to set the DNS or do any workarounds when you want to run things locally.

Serving using HTTPS (green padlock)

Just set the port to 443, which is the standard for HTTPS. Static Sites will handle obtaining any certificates, caching them, and renewing them (for all the sites).

The first time a site is hit there may be a slight delay whilst certificates are obtained, but subsequent hits will automatically use the cached version and not be subject to that brief pause.

Things to note

  • The site folders as specified in the sites.ini file should either be relative to the folder you launch the server from, or fixed to your home folder (eg ~/sites), or absolute paths that will work from anywhere.
  • The number of sites you can serve is limited by the number of host names LetsEncrypt will allow in their certificates (around 100).

Simplicity is the driving force. You get configuration free serving of all your static sites using HTTPS with the smallest effort possible from a self-hosted facility. Requests for things such as Markdown processing, folder listings and so forth are beyond that scope.
