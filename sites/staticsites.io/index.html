About Static Sites

  • Run all your static websites on a single port with just one server.
  • Have your sites running HTTPS with absolutely zero configuration.
  • No need for a traditional web server deploy, meaning tiny resource usage.
  • Production ready with a single file deployment and great performance.
  • Open source codebase for peace of mind: view the code on my GitHub.

Installing

Whether you are wanting to run on your local machine or a server/cloud instance makes no difference. Follow the steps below to get started.

Download the relevant version

  • Download the Mac OS version
  • Download the Linux version
  • Download the Windows version

Check the dependencies and install it

There are no dependencies. There is no installation needed.
Just drop the download into a suitable folder and it is ready to run.

Running

Copy your static site(s) into place in folders alongside the downloaded application, named as per the domains. Remember that on Linux servers upper/lower case matters. The folder tree should look something like this: 

/home/karl/my-sites
    StaticSites
    /staticsites.io
        index.html
    /www.example.com
        index.html

You can then run the server: 

cd /home/karl/my-sites
./StaticSites -sites ./sites -port 8000 -local staticsites.io

This will launch all sites found in the ./sites folder on port 8000 (which is the default anyway, but shown here for completeness). The sites will all start responding to requests to their domain name provided of course that your DNS routes them in.

Don't expect to see any sites responding at localhost by default, as the routing is done by domain name so you would usually need to browse to the domain. In order to make things easier for local work, the local parameter allows you to nominate one site to run as localhost so there is no need to set the DNS or do any workarounds when you want to work on it as it will then respond directly to localhost.

Serving using HTTPS (green padlock)

Just set the port to 443, which is the standard for HTTPS. Static Sites will handle obtaining any certificates, caching them, and renewing them (for all the sites).

The first time a site is hit there may be a slight delay whilst the certificate is updated, but subsequent hits to that site will automatically use the cached version and not be subject to that brief pause.

Notes

  • Sites are scanned at startup, so adding/removing a site requires a StaticSites restart. Changing the content of a site folder does not.
  • The site folders should be the complete domain name, including subdomain, but without any protocol, like example.com or blog.kcartlidge.com.
  • The site folders should be accessible to the relevant user if you have the server launch automatically (eg with pm2 or supervisor) and it runs under a different user.
  • The number of sites you can serve is limited by the number of host names LetsEncrypt will allow in their certificates (around 100), as a single certificate is maintained for all the domains. There are also rate limits applied by LetsEncrypt for certificate issuance, but certificates are only requested when port 443 is used so simply avoid that port until ready.
  • No caching is currently implemented. Static sites are extremely light on resource usage, and the Static Sites server itself hugely lighter than fully fledged traditional web servers, so performance should not be an issue. If it is, static sites are also prime candidates for services like CloudFlare or equivalent.

Simplicity is the driving force. You get configuration free serving of all your static sites using HTTPS with the smallest effort possible from a self-hosted facility. Requests for things such as Markdown processing, folder listings and so forth are beyond that scope.
